# 회고록 — 중소기업 AI 도입 컨설팅 신청 서비스

> ✍️ 이 문서는 초안입니다. **굵게 표시된 개인 소감 부분**은 본인의 말로 다듬어주세요. 나머지는 실제로 진행한 내용을 정리한 것입니다.

## 무엇을 만들었나 (한 줄)

중소기업 대표가 **로그인 없이** 이메일·휴대폰번호만으로 "AI 도입 컨설팅"을 신청하고, **매주 4명 정원**을 절대 넘기지 않으며, 로그인하면 내 신청을 편하게 확인하는 웹 서비스를 스펙부터 배포까지 만들었다.

## 왜 이 문제였나 (도메인 연결)

반도체 엔지니어를 거쳐 해외 통신망 설치회사를 **직접 운영**하며, 중소기업 대표가 모든 걸 혼자 짊어지는데 인적·물적 자원은 늘 부족한 현실을 겪었다. AI 도입이 그 격차를 메우는 가장 현실적인 방법이라 확신했고, "컨설팅 신청"이라는 가장 낮은 문턱의 첫걸음을 서비스로 만들었다.

## 오늘 결정한 것 (그리고 왜)

1. **스펙 먼저(OpenSpec)** — 코드보다 "무엇을 만들지"를 먼저 글로 확정. 덕분에 중간에 헤매지 않았다.
2. **기술 스택 = FastAPI + SQLite** — 파이썬 친숙도 + 배포 단순함. 후보(Flask, Node)와 비교해 근거를 문서(design D0)에 남겼다.
3. **"로그인 없이 신청"을 끝까지 유지** — 대표들의 진입장벽을 낮추는 게 핵심 가치라, 로그인은 나중에도 "선택"으로만 얹었다.
4. **주 4명 정원을 트랜잭션으로 보장** — 동시에 신청이 몰려도 절대 초과되지 않도록 DB 트랜잭션으로 처리. 이걸 테스트(동시 10명 → 정확히 4명)로 증명했다.
5. **Supabase는 '인증만', 신청 데이터는 SQLite 유지(하이브리드)** — 변경 범위를 최소화하려는 결정.

## 오늘 참은 것 (안 한 것, 그리고 왜)

- **신청 데이터를 Supabase(PostgreSQL)로 옮기지 않았다** — 이번 범위를 넘어서고 복잡해지므로 다음으로 미룸.
- **대기(waitlist)·관리자 페이지·알림(SMS/메일)·소셜 로그인** — 핵심 가치와 거리가 있어 의도적으로 제외.
- **정원 4명을 설정으로 바꾸는 기능** — 지금은 상수 한 곳으로 고정, 나중에 쉽게 바꿀 수 있게만 준비.
- **화면을 과하게 꾸미지 않았다** — 신뢰감 있는 최소 디자인에서 멈춤.

## AI와 협업하며 배운 점

- **"무엇을"과 "어떻게"를 나눠 말하니** 결과가 정확해졌다 — 요구사항을 먼저 확정(스펙)하고 구현을 맡기는 흐름이 효율적이었다.
- **애매한 지점은 AI가 결정지어 문서에 남겨두니** 나중에 근거를 되짚기 좋았다(예: "주"의 기준, 중복의 범위).
- **비밀값(.env)은 내가 직접 넣고 AI는 코드만** — 역할을 나누니 안전하게 진행됐다.

## 어려웠던 점 (실제로 막혔던 것들)

- **배포 직후 접속했더니 404가 떴다** — 알고 보니 무료 서버가 잠들었다 깨어나는(cold start) 중이었고, 로그를 보니 앱은 정상이었다. → *로그를 볼 줄 아는 게 중요*하다는 걸 배웠다.
- **로그인이 안 됐다** — Supabase의 "이메일 확인(Confirm email)"이 켜져 있어서였다. 설정을 끄고, 새 계정으로 가입하니 해결.
- **`.env.example`이 실수로 무시**됐다 — `.gitignore` 패턴이 너무 넓어서. 예외 규칙을 추가해 해결.

## 오늘 도달한 지점

계획한 **5단계(실제로는 0~9단계 전부)** 를 모두 굴렸다: 스펙 → 저장소 → PRD → 뼈대 → 화면 → 디자인 → 저장/정원 → 배포 → 로그인. 테스트 21개 통과, 배포 URL 동작 확인 완료.
