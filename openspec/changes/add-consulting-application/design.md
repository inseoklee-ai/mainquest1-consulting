## Context

중소기업 대표 대상 "AI 도입 컨설팅 신청" 기능이다. 로그인이 없고, 개인정보(이메일·휴대폰번호)를 다루며, **주당 4명 정원을 절대 넘기면 안 되는** 제약이 핵심이다. 백엔드 스택은 아래 **D0** 에서 후보를 비교해 **Python + FastAPI + SQLite** 로 선정했다. (동작 요구사항은 `specs/consulting-application/spec.md` 참조)

## Goals / Non-Goals

**Goals:**
- 로그인 없이 이메일+휴대폰번호로 신청/조회
- 동일인(이메일+휴대폰 조합) 전체 1회 신청 보장
- 주(월~일, KST) 단위 정원 4명을 **동시성 상황에서도** 초과하지 않도록 보장
- 초보 학습자가 이해·구현할 수 있을 만큼 단순한 구조

**Non-Goals:**
- 회원가입/인증 시스템 (이번 범위 아님)
- 관리자 대시보드, 결제, 알림(이메일/SMS 발송) — 이번 범위 아님
- 정원을 관리자가 바꾸는 기능 — 이번엔 4명 고정(D 참조)

## Decisions

### D0. 백엔드 스택 선정 → **Python + FastAPI + SQLite**

**후보 비교**

| 항목 | Python + FastAPI + SQLite ✅ | Python + Flask + SQLite | Node.js + Express + SQLite |
|------|------------------------------|--------------------------|-----------------------------|
| 학습자 친숙도 | 높음 (파이썬 중심 부트캠프, Todo 앱서 SQLite 경험) | 높음 | 중간 (JS 별도 학습 필요) |
| 입력 검증(이메일/휴대폰 형식) | **Pydantic 내장** — 코드 적고 안전 | 수동 구현 필요 | 라이브러리(zod 등) 추가 필요 |
| 자동 API 문서 | **Swagger UI 자동 생성** | 없음(별도 설치) | 없음(별도 설정) |
| 정원 동시성 제어 | SQLite 트랜잭션(`BEGIN IMMEDIATE`)으로 가능 | 동일 | 동일 |
| 배포 난이도 | 낮음 (DB가 파일 1개) | 낮음 | 낮음 |
| 학습 자료/한글 자료 | 풍부 | 매우 풍부 | 풍부 |

**결정 근거 (왜 FastAPI + SQLite인가)**
- **학습자 배경 적합**: 파이썬 중심 학습 경로 + Todo 앱에서 이미 SQLite를 다뤄봤으므로 러닝커브가 가장 낮다. 새 기능 로직에 집중할 수 있다.
- **이 앱의 핵심 요구와 궁합**: 이메일·휴대폰 **형식 검증**이 요구사항인데, FastAPI의 **Pydantic**으로 타입·형식 검증을 선언적으로 처리할 수 있어 코드가 짧고 실수가 준다.
- **정원 동시성(D3) 충족**: 주 4명이라는 소규모 트래픽에서는 SQLite의 쓰기 트랜잭션 직렬화(`BEGIN IMMEDIATE`)만으로 "정원 초과 절대 불가"를 안전하게 보장할 수 있다.
- **배포 학습에 유리**: SQLite는 DB 파일 하나면 되어 배포·백업이 단순하다. 배포까지 실습하려는 목표에 부담이 적다.
- **자동 문서(Swagger)**: FastAPI가 API 문서를 자동 생성해, 신청/조회 엔드포인트를 브라우저에서 바로 눌러보며 학습·검증하기 좋다.

**왜 다른 후보는 아닌가**
- **Flask**: 훌륭하지만 입력 검증·문서화를 직접 붙여야 해 초보에겐 손이 더 간다. FastAPI가 같은 파이썬이면서 이 부분을 기본 제공.
- **Node.js + Express**: 좋은 선택지지만, 파이썬 대비 새로 익힐 것이 많아 이번 학습 목표(백엔드+배포 흐름 체득)에서 인지 부하가 커진다.

**한계와 대비책**
- SQLite는 **다중 서버 인스턴스**나 **고동시성**에는 약하다. 이 앱 규모(주 4명)에선 문제없으나, 훗날 트래픽이 커지면 **PostgreSQL로 이전**을 고려한다(데이터 모델 D4는 그대로 이식 가능하도록 표준 SQL 위주로 설계).

### D1. "주" 판정 기준
- **결정**: 신청 **접수 시각**을 KST로 변환 → 그 시각이 속한 **월요일~일요일** 주를 "신청 주"로 본다.
- **저장 방법 권장**: 신청 레코드에 `week_key`(예: `2026-W30` 또는 그 주 월요일 날짜 `2026-07-20`)를 함께 저장해두면, 정원 카운트가 단순 `COUNT(*) WHERE week_key = ?` 로 끝난다.
- **대안**: 신청 시각만 저장하고 조회 시마다 주 범위를 계산 → 쿼리가 복잡하고 실수 여지 큼. 그래서 `week_key` 를 명시 저장하는 쪽을 택함.

### D2. 동일인 판별 & 중복 차단
- **결정**: (이메일, 휴대폰번호)를 정규화(이메일 소문자화, 휴대폰 숫자만 추출) 후 이 조합에 **유니크 제약(UNIQUE index)** 을 건다.
- **이유**: 애플리케이션 코드에서 "먼저 조회 후 없으면 저장"만 하면 동시 요청에서 둘 다 통과할 수 있다. DB 유니크 제약이 최종 방어선.

### D3. 정원 초과 방지(동시성) — 가장 중요
- **결정(선정 스택 기준, D0)**: **SQLite** 의 쓰기 트랜잭션은 사실상 직렬화되므로 이를 활용한다.
  `BEGIN IMMEDIATE` 로 트랜잭션 시작 → `COUNT(*) WHERE week_key=?` 확인 → 4 미만이면 INSERT → COMMIT.
  (훗날 PostgreSQL로 이전 시엔 트랜잭션 안에서 주별 카운터 행을 `SELECT ... FOR UPDATE` 로 잠그는 방식으로 대체)
- **대안(비권장)**: 애플리케이션 메모리에서 카운트 관리 → 서버 재시작·다중 인스턴스에서 깨짐.
- **검증 포인트**: spec의 "동시 신청이 정원을 넘지 않음" 시나리오를 반드시 테스트로 확인.

### D4. 데이터 모델 (SQLite 기준, D0)
`applications` 테이블:
| 필드 | 설명 |
|------|------|
| id | 기본키 |
| email | 정규화된 이메일 |
| phone | 정규화된 휴대폰번호(숫자만) |
| week_key | 신청이 속한 주 식별자 |
| created_at | 신청 접수 시각(UTC 저장, 표시/판정은 KST) |
- 제약: `UNIQUE(email, phone)` — 중복 차단(D2)
- 인덱스: `week_key` — 정원 카운트(D1, D3)

### D5. 인터페이스(개념)
- 신청 접수: 입력(email, phone) → 결과(접수 성공 / 중복 / 정원 마감 / 형식 오류)
- 내 신청 내역 조회: 입력(email, phone) → 두 값이 모두 일치하는 신청만 반환(D2 정규화 동일 적용)
- 형태(웹 폼 + 서버 API 등)는 스택 확정 후 결정.

## Risks / Trade-offs

- **[개인정보 노출]** 이메일만 알아도 조회되면 안 됨 → 조회는 반드시 이메일+휴대폰 **둘 다 일치** 조건 사용(spec의 "타인 내역 비노출" 시나리오).
- **[동시성 오구현]** "조회 후 삽입"만으로는 정원이 새어나갈 수 있음 → D3의 트랜잭션/락 + D2의 유니크 제약을 함께 사용.
- **[시간대 실수]** 서버가 UTC로 돌면 주 경계가 어긋날 수 있음 → 주 판정은 항상 KST 기준으로 계산(D1).
- **[정원 고정]** 4명 고정이라 운영 중 변경이 어려움 → 지금은 학습 범위상 상수로 두되, 코드 한 곳(상수)에 모아 나중에 설정화하기 쉽게 둔다.

## Migration Plan

신규 기능이라 마이그레이션 이슈 없음. 배포 시 `applications` 테이블 생성 + 유니크 제약/인덱스 적용이면 충분.
