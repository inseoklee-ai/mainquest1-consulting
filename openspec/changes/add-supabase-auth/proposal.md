## Why

지금은 로그인 없이 이메일+휴대폰번호만으로 신청/조회한다. 이 방식은 진입장벽이 낮아 좋지만, 재방문한 사용자는 매번 두 값을 다시 입력해야 "내 신청"을 볼 수 있다. **원하는 사용자에게 선택적으로 회원가입/로그인을 제공**하면, 로그인한 사람은 재입력 없이 자신의 신청 내역을 편하게 확인할 수 있다. 인증 기능을 직접 구현하면 복잡하고 보안 위험이 크므로 **Supabase Auth**를 빌려 쓴다.

## What Changes

- **선택적 회원가입/로그인 추가** (Supabase Auth, 이메일+비밀번호). 로그인은 **강제가 아니다.**
- 로그인/로그아웃/회원가입 화면 및 세션 처리 추가
- **로그인 사용자의 "내 신청" 조회**: 이메일+휴대폰 재입력 없이, 로그인 계정의 검증된 이메일로 신청 내역 표시
- 기존 **"로그인 없이 신청/조회" 흐름은 그대로 유지** (비로그인 조회는 계속 이메일+휴대폰 둘 다 필요)

## Decisions

- **D1. 인증 제공자**: **Supabase Auth (이메일+비밀번호)**. 직접 구현 대비 보안·구현 부담이 적고 학습용으로 자료가 풍부.
- **D2. 하이브리드 저장**: **인증(계정)만 Supabase**, **신청 데이터는 기존 SQLite 유지.** 신청 테이블 구조는 바꾸지 않는다. (변경 범위 최소화)
- **D3. 로그인 사용자의 "내 신청" 매칭 기준 = Supabase가 검증한 이메일.**
  - 비로그인 조회: 이메일+휴대폰 **둘 다** 일치 필요 (본인 확인 수단이 그것뿐이므로).
  - 로그인 조회: 이메일이 **이미 인증**되었으므로, 그 이메일과 일치하는 신청을 보여준다(휴대폰 재입력 불필요).
  - 근거: 로그인은 이메일 소유를 증명하므로, 같은 수준의 본인 확인이 이미 이루어진 것으로 본다.
- **D4. 비밀값 관리**: `SUPABASE_URL`, `SUPABASE_ANON_KEY` 등은 **`.env`로 관리하고 저장소에 커밋하지 않는다.** (`.gitignore`에 `.env` 이미 포함)
- **D5. 세션/토큰 처리 방식**은 design에서 확정(서버 세션 쿠키 vs Supabase 토큰 저장).

## Capabilities

### New Capabilities
- `user-auth`: Supabase 기반 회원가입/로그인/로그아웃 및 세션 관리.

### Modified Capabilities
- `consulting-application`: "내 신청 내역 조회" 요구사항 확장 — 로그인 사용자는 검증된 이메일 기준으로 조회 가능(휴대폰 재입력 없이). 비로그인 조회 동작은 그대로.

## Impact

- **신규 의존성**: Supabase (supabase-py 또는 REST), 세션 처리 라이브러리(필요 시)
- **신규 환경변수**: `SUPABASE_URL`, `SUPABASE_ANON_KEY` (`.env`, 커밋 금지)
- **화면**: 로그인/회원가입/로그아웃 UI, 헤더에 로그인 상태 표시
- **기존 코드**: `/my` 조회 로직에 "로그인 시 이메일 기준" 분기 추가. 신청 저장 로직·SQLite 스키마는 변경 없음.
- **배포**: Render 환경변수에 Supabase 키 등록 필요 (docs/DEPLOY.md 갱신 대상)
- **보안/개인정보**: 토큰·비밀값 노출 금지, 비로그인 타인 내역 비노출 규칙 유지.
