## ADDED Requirements

### Requirement: 회원가입

시스템은 방문자가 이메일과 비밀번호로 계정을 만들 수 있어야 한다(SHALL). 계정 생성은 Supabase Auth를 통해 처리한다. 로그인은 서비스 이용의 **필수 조건이 아니다.**

#### Scenario: 정상 회원가입
- **WHEN** 방문자가 올바른 이메일과 비밀번호 정책을 만족하는 비밀번호로 가입을 요청한다
- **THEN** 시스템은 Supabase에 계정을 생성하고, 가입이 완료(또는 이메일 확인 안내)되었음을 알린다

#### Scenario: 이미 존재하는 이메일로 가입
- **WHEN** 이미 계정이 있는 이메일로 다시 가입을 요청한다
- **THEN** 시스템은 가입을 거부하고, 이미 존재하는 계정임을 안내한다

#### Scenario: 잘못된 입력
- **WHEN** 이메일 형식이 틀리거나 비밀번호가 정책에 미달한다
- **THEN** 시스템은 가입을 거부하고, 무엇을 고쳐야 하는지 안내한다

### Requirement: 로그인 / 로그아웃

시스템은 등록된 사용자가 이메일+비밀번호로 로그인하고, 로그아웃할 수 있어야 한다(SHALL). 로그인 성공 시 세션이 유지되어 이후 요청에서 본인으로 식별된다.

#### Scenario: 정상 로그인
- **WHEN** 등록된 사용자가 올바른 이메일과 비밀번호로 로그인한다
- **THEN** 시스템은 세션을 생성하고, 로그인 상태로 전환한다

#### Scenario: 잘못된 자격 증명
- **WHEN** 존재하지 않는 계정이거나 비밀번호가 틀린 상태로 로그인한다
- **THEN** 시스템은 로그인을 거부하고, 자격 증명이 올바르지 않음을 안내한다(어느 쪽이 틀렸는지는 구분하지 않는다)

#### Scenario: 로그아웃
- **WHEN** 로그인한 사용자가 로그아웃한다
- **THEN** 시스템은 세션을 종료하고, 비로그인 상태로 전환한다

### Requirement: 로그인 상태 표시

시스템은 현재 로그인 여부를 화면에 표시해야 한다(SHALL). 로그인 상태에 따라 제공되는 동작(로그인/로그아웃 링크 등)이 달라진다.

#### Scenario: 로그인 상태
- **WHEN** 로그인한 사용자가 페이지를 연다
- **THEN** 헤더에 로그인된 이메일(또는 로그아웃 링크)이 보인다

#### Scenario: 비로그인 상태
- **WHEN** 로그인하지 않은 방문자가 페이지를 연다
- **THEN** 헤더에 로그인/회원가입 링크가 보이고, 기존의 로그인 없는 신청/조회는 그대로 가능하다

### Requirement: 비밀값 보호

시스템은 Supabase 접속 비밀값(URL, 키)과 세션 토큰을 안전하게 다뤄야 한다(MUST). 비밀값은 소스 저장소에 커밋하지 않는다.

#### Scenario: 비밀값 미노출
- **WHEN** 애플리케이션이 배포/실행된다
- **THEN** Supabase 키·토큰은 환경변수(.env)에서 읽으며, 클라이언트에 불필요하게 노출되거나 저장소에 커밋되지 않는다
